# L2/26-062

## 要約

`L2/26-062` は、Indian languages の text rendering、input、search、processing に関する実装上の問題と recommendation をまとめた文書である。

主対象は Brahmi-derived scripts で、例は Devanagari が中心だが、Perso-Arabic script を使う Urdu / Kashmiri / Sindhi などにも触れる。

Rendering では、atomic vowel letters / signs をユーザが分解入力した場合や、dependent marks の重複・順序違いを、他の valid sequence と同じ見た目にしないことを重視する。

Input、search、NLP preprocessing では、language-specific valid character set、orthography validation、systematic spelling variation の folding / normalization を扱う。

## 報告内容

### Text rendering

Atomic に符号化された vowel letters / signs を、分解 sequence で入力した場合には invalid とし、同じ glyph に見せないことを提案する。Dependent marks が複数ある sequence は、Unicode Standard Section 5.13 の inside-out rule に従い、invalid な重複や順序違いは dotted circle で表示するべきだとする。

ただし、すべての unusual sequence を dotted circle にするのではなく、他の sequence と confusable にならないものは expression の自由を保つため dotted circle なしで表示してよい、と整理している。

### Text input

Language users は script 全体では valid でも、特定 language では使わない letters / sequences を、visual confusability や keyboard unfamiliarity により入力し得る。文書は Enhanced InScript layouts を context-sensitive に拡張し、同じ key が前文脈に応じて independent vowel / dependent vowel sign を出し分ける "Contextual Keyboard Layout" を提案する。

### Text search と processing

Search では、Hindi の anusvāra / pancamākṣara variation のような systematic spelling variation を、case folding や `ß = ss` と同様に search / collation rules へ組み込むことを提案する。

Processing では、NLP preprocessing として language-specific orthography validation を行い、必要に応じて spelling variation や cross-script forms を normalization する方向を示す。

## 後続決定

[L2/26-138](utc-l2-26-138.md) は consecutive anusvaras の rendering issue を扱うとき、この文書を background として参照している。`L2/26-062` 自体は encoding proposal ではなく、Indic rendering / input / validation の設計観点を提供する report として読む。

## 論点

### invalid sequence と dotted circle の標準化

文書は、invalid sequence を dotted circle で表示する条件をより標準化する必要を示す。これは font shaping の実装差だけでなく、phishing / spoofing のような confusable rendering の抑止にも関わる。

### language-specific validation

IS16350:2016 の script / language keyboard data を使って、各 language で valid な code points / sequences を検証するという発想は、Unicode property だけではなく locale / language layer と連携する。
