# L2/26-107

## 要約

`L2/26-107` は、Proposed Update UAX \#57 Revision 6 "Unicode Egyptian Hieroglyph Database (Unikemet)"、Unicode 18.0.0 draft 3 である。

UAX \#57 は、Egyptian hieroglyphs の ancillary data を `Unikemet.txt` としてどう格納し、catalog indexes、sources、descriptions、function、Core set、mirroring / rotation、alternate sequences をどう読むかを説明する。

特に Egyptian Hieroglyphs Extended-A では algorithmic character names だけでは character identity が分かりにくいため、Unikemet data が実装者と Egyptologists のための識別・検索・font support の入口になる。

## 提案内容

### Unikemet.txt の構造

`Unikemet.txt` は UCD に含まれる UTF-8 / NFC / Unix line endings の text file であり、code point、property name、property value の tab-separated fields を持つ。comments は `#` で始まり、同じ hieroglyph に同じ property は複数回出ない。

### property categories

文書は property を、catalog indexes、sources、description、function、Core、mirroring / rotation、other mappings に分類する。

| 分類 | 主な property | 役割 |
| --- | --- | --- |
| catalog indexes | `kEH_Cat`, `kEH_UniK` | Gardiner / IFAO / Unikemet 系の識別子を統合する。 |
| sources | `kEH_HG`, `kEH_JSesh`, `kEH_IFAO` | Hieroglyphica、JSesh、IFAO entries などの source references。 |
| description | `kEH_Desc` | sign の形状記述。informative だが identity の理解に重要。 |
| function | `kEH_Func`, `kEH_FVal` | pictogram、logogram、phonogram、classifier などの機能分類と値。 |
| Core | `kEH_Core` | Core / Legacy / None の curated status。 |
| orientation | `kEH_NoMirror`, `kEH_NoRotate` | 通常と異なる mirroring / rotation behavior。 |
| alternate sequences | `kEH_AltSeq` | encoded sign と Egyptian format controls からなる alternate sequence。 |

### Core set と evidence

`kEH_Core` は、専門家が evidence を確認した Core set を表す。Core set の判定では photographs や trustworthy facsimiles が重視され、transcription だけでは通常 Core evidence として扱われない。

## 後続決定

[L2/26-095](utc-l2-26-095.md) は PRI \#538 として UAX \#57 proposed update を public review issue に含める。[L2/26-096](utc-l2-26-096.md) は `kEH_FVal` の delimiter を `U+007C` に統一し、`/` との併記をやめる recommendation を記録する。

## 論点

### algorithmic names を補う data file

Extended-A の algorithmic names は code point 以上の識別情報をほとんど持たないため、Unikemet の catalog / source / description properties は、font、search、学術参照で character identity を扱う基盤になる。

### normative と provisional の区別

source properties は normative な database content とされる一方、function や Core などには provisional な側面が残る。実装で使う場合は、property ごとの stability と update risk を分けて見る必要がある。
