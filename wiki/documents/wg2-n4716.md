# WG2 N4716

## 要約

`WG2 N4716` は、Suzuki Toshiya が `Shuowen Jiezi` の版本ごとの Small Seal glyph を source-based variation selector で表現するよう提案した individual contribution である。`L2/16-092` は同一 PDF を UTC register に載せた別番号なので、このページでは alias として扱う。

文書は、[WG2 N4688](wg2-n4688.md) が主な source を Tenghuaxie version (`THX`) に絞った一方、他版本の形が異なる場合に別 code point を割り当てる方向を問題視する。glyph shape だけによる別符号化は、版本間の情報交換と既存 font の移行を難しくすると論じる。

代案は、character identity を `Shuowen Jiezi` の記述に基づいて共有し、`THX`、`PJG`、`CCZ`、`DYC` などの版本差を source-based variation sequence で選択する方式である。ただし、同形に近い重複 entries の扱いは variation selector だけでは解決できず、追加調査が必要だとする。

## 提案内容

| 項目 | 内容 |
| --- | --- |
| 基本モデル | Small Seal character の identity と、`Shuowen Jiezi` 各版本の glyph representation を分離する。 |
| selector の意味 | shape category ではなく、`THX`、`PJG`、`CCZ`、`DYC` など source version を指定する。 |
| 対象となる差 | 微細で不安定な差、構造上明確だが記憶しにくい差、版本全体で体系的な差の三類型を検討する。 |
| 実装例 | OpenType `cmap` format 14 を使う sample font で、base character に版本別 variation sequence を適用する。demo の selector 値は正式な割当提案ではない。 |
| 期待される効果 | 版本名を手掛かりに variant を選べるようにし、shape-based な別 code point の増加と既存 font の非互換化を避ける。 |

文書は、glyph 差を Type A（crossing / touching など微細で一貫性を見出しにくい差）、Type B（IDS では区別できるが character ごとに不規則な構造差）、Type C（DYC の `門` component のように版本内で体系的な差）に分ける。Type B / C は registry 化できる可能性がある一方、Type A の unification boundary は安定していないと指摘する。

## 後続決定

この文書自体は proposal であり、会議決定ではない。TCA は後続の `WG2 N4755` で N4716 に feedback を提出した。2019 年の [WG2 N5108](wg2-n5108.md) は WG2 \#68 の discussion として、Zhuanwen / Guwen / Zhouwen の基本的な disunification、非常に似た glyph の unification、他版本と taxonomy の追加検討を報告している。

最終的な後続提案は単純な source-based VS model に一本化されず、複数版本を source columns として同一 database / code chart に保持しながら unification / disunification を判断する方向へ進んだ。[WG2 N5307R](wg2-n5307r.md) は版本間の mapping principles を整理し、[WG2 N5344R2](wg2-n5344r2.md) は `THX`、`CCZ`、`QJZ`、`DYC` の four-column source model を採る revised proposal である。

## 論点

### Character identity と glyph source

N4716 の中心的な問題提起は、版本ごとの形をそのまま別 character とみなすと、利用者が同じ `Shuowen Jiezi` entry を検索・交換する際に code point を選べなくなることである。文書は shape よりも text description を安定した identity の手掛かりとするが、説明も形もよく似た重複 entries についてはこの方法でも曖昧さが残ると認めている。

### Representative glyph と既存実装

文書は `THX` を representative glyph の唯一の基準とする根拠が十分でないと問い、他の Song dynasty 系資料との関係や版本評価が確定していないことを挙げる。また、`THX` と異なる既存 Small Seal fonts に別 code point への移行を強いることは、標準化による相互運用性を損なうと懸念する。
